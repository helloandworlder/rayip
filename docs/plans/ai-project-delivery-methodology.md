# AI 项目规划与编码交付方法论

> 适用范围：任意软件项目的 AI 辅助规划、重构、从 0 到 1 开发、复杂需求拆解。  
> 核心原则：业务目标决定重要性，技术风险决定先后顺序，垂直切片决定 Task 形状，验收标准决定是否完成。

## 1. 基本判断

AI 编码项目不能按“后端、前端、数据库、部署”这种横向层拆计划。

正确拆法是：

- 先明确项目目标和 V1 范围。
- 再识别最可能推翻项目的技术风险。
- 然后把工作拆成可运行、可验收的垂直切片。
- 每个 Task 完成后，都应该多出一组真实可用的产品能力。

## 2. 项目章程

每个项目开始前先写清楚：

- 产品目标。
- 主要用户。
- V1 做什么。
- V1 不做什么。
- 商业或业务成功指标。
- 最大技术风险。
- 最大业务正确性风险。
- 发布前验收门槛。

章程要短，只服务于取舍，不写成论文。

## 3. 风险前置

风险排序规则：

- 如果这个问题失败，项目是否失败？
- 如果晚发现，是否会推翻大量已写代码？
- 如果不先验证，后续设计是否都可能错？

答案是“是”的风险必须前置。

常见例子：

- 代理平台：先验证 NodeAgent、Runtime 控制、限速、连接数、恢复，再做完整钱包和订单。
- AI 产品：先验证模型效果、延迟、成本、评测，再做完整页面。
- 支付产品：先验证账本、幂等、对账，再做营销页。
- IoT 产品：先验证设备通信、掉线恢复，再做后台 CRUD。

## 4. Milestone 设计

推荐 2-4 个 Milestone。

通用结构：

| Milestone | 目标 |
|---|---|
| M1：可行性闸门 | 验证最大技术风险，证明项目能成立 |
| M2：核心产品闭环 | 交付用户能完成的主流程 |
| M3：生产运营闭环 | 交付管理、观测、恢复、发布硬化 |
| M4：规模化增强，可选 | 扩展商业化、容量、自动化能力 |

不要按技术层设计 Milestone：

- 不要 `M1 后端`。
- 不要 `M2 前端`。
- 不要 `M3 数据库`。
- 不要 `M4 部署`。

这种拆法会隐藏集成风险。

## 5. 厚 Task

一个好的 Task 应该包含：

- 用户或管理员入口。
- API 或服务行为。
- 数据模型和关键约束。
- 异步任务或外部集成，如果相关。
- 核心领域逻辑。
- 管理端可见性、审计或观测。
- 测试或可执行验证。

如果一个 Task 只是：

- 实现后端。
- 创建表。
- 做页面。
- 接入队列。
- 写某个模块。

它通常太薄，不是产品交付 Task。

## 6. Task 模板

```markdown
### Tn：任务名称

价值：
- 完成后用户、管理员或运营人员能做什么？

为什么现在做：
- 这个 Task 先解决哪个风险？
- 哪些后续能力依赖它？

交付范围：
- 前端：
- API / 服务：
- 数据：
- 异步任务 / 集成：
- Runtime / 外部系统：
- 管理端可见性：
- 测试：

验收：
- [ ] 可观察的成功结果。
- [ ] 明确的失败行为。
- [ ] 幂等 / 并发 / 恢复检查，如果相关。
- [ ] 用户或管理员能验证结果。
```

## 7. 推荐排序

默认顺序：

1. 薄工程基线：能启动、能构建、能测试。
2. 最大技术可行性风险：用可见工具或测试台验证。
3. 可靠数据或任务链路：证明状态、队列、重试、幂等成立。
4. 真实可用性闸门：证明可售、可用、可交付不是手填状态。
5. 第一个用户主流程：完成核心业务闭环。
6. 生命周期和异常处理：续费、停用、失败、恢复、对账。
7. 管理运营和观测：让管理员能判断、追踪、修复。
8. 发布硬化：部署、故障演练、备份、回滚、安全检查。

一句话：

> 先证明能不能做，再证明能不能稳定做，再证明用户能不能用，最后证明管理员能不能运营。

## 8. Definition of Done

Task 未满足以下条件，不算完成：

- 有可见入口或命令路径。
- 关键状态有根源持久化。
- 失败行为明确。
- 有验证证据。
- 计划状态已更新。

商业系统还必须满足：

- 不重复扣款。
- 不重复分配库存。
- 不在履约确认前发放权益。
- 管理员能看到卡住的工作。

## 9. SubAgent 审查模板

重要计划需要独立审查。

```text
请独立审查当前项目规划，不要修改文件。

重点判断：
1. 最大技术风险是否足够前置？
2. Task 是否是垂直切片，而不是技术层拆分？
3. 每个 Task 完成后是否有可运行、可验收的产品能力？
4. 业务正确性风险是否在发布前覆盖？
5. 哪些 Task 太薄、太宽或顺序错误？
6. 如果需要，请给出更好的 Milestone / Task 拆分。
```

审查结果只在能提升计划质量时合并。

## 10. RayIP 当前应用

RayIP 的最大风险不是钱包、用户、产品 CRUD，而是：

- NodeAgent 是否能稳定连接控制面。
- 魔改 XrayCore 是否能账号级增量管理。
- XrayCore 内限速、连接数、流量统计是否可信。
- gRPC 增量 apply 是否幂等。
- PG outbox + NATS + Worker 是否能可靠下发。
- 单节点 10k+ 账号恢复是否不依赖整份配置重写。

因此 RayIP 的计划采用 Runtime-first：

- M1：Runtime 可行性闸门。
- M2：商业交易闭环。
- M3：生产运营闭环。

这不是把商业放后面，而是避免把商业流程建在不确定的数据面上。

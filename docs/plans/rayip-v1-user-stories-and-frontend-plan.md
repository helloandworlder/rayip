# RayIP V1 用户故事与前端设计计划

> 日期：2026-04-30  
> 范围：用户面板、管理面板、实时节点面板  
> 参考：`frontend_design/` 下 12 张 RocketIP / IPIPD 风格截图

## 1. 目标

本文档把 RayIP V1 的前端体验和用户故事固定下来，作为后续实现 `apps/user-web` 和 `apps/admin-web` 的验收依据。

核心目标：

- 客户注册、充值、购买、查看代理、续费、刷新 IP、刷新凭据、看流量、提交反馈必须顺畅。
- 管理员管理产品、库存、节点、订单、任务、通知、Web SSH 必须高效。
- 客户只看到产品化结果，不看到节点下发、Runtime apply、任务重试等内部概念。
- 管理端可以看到内部状态，但它是运营和排障工具，不是客户下单失败后的兜底列表。

## 2. 设计参考约束

`frontend_design/` 是 RayIP 用户面板第一版的硬参考。可以替换品牌、文案和数据模型，但布局节奏、信息密度和核心交互必须对齐。

需要保留的体验特征：

- 左侧分组导航：概览、静态住宅代理、工具、账单、推广、帮助和支持。
- 顶部操作区：账户余额、充值、主题切换、语言、账户菜单。
- 页面背景浅灰，内容区域白底，主色蓝色，强调清晰和速度。
- 购买页是核心工具页：选择用途、IP 类型、地区、线路、协议、时长、数量，右侧或固定区域显示订单摘要。
- 已购 IP 列表是高频工作台：筛选、复制、批量续费、刷新 IP、刷新凭据、导出。
- API 密钥页演进为开发者 App 页：多个 App、`AppID + AppSecret`、IP 白名单、回调地址、重置 Secret、调用统计。
- 代理验证工具保留：客户可以粘贴 HTTP/SOCKS5 代理并验证可用性。
- 充值、计费、优惠券、推广、反馈页面保持简洁表单和表格，不做复杂客服系统。

设计禁止项：

- 不做泛 SaaS 营销大屏作为登录后首页。
- 不在用户面板暴露节点、任务、下发失败、Runtime、NATS、Worker 等内部术语。
- 不使用大面积渐变、装饰插画、过大圆角、卡片套卡片。
- 不用“可稍后修复”的状态替代购买前可售判断。

## 3. 用户面板信息架构

建议路由：

| 路由 | 页面 | V1 能力 |
|---|---|---|
| `/login` | 登录 | 邮箱/手机号登录、找回密码入口 |
| `/register` | 注册 | 注册、服务条款确认 |
| `/dashboard` | 概览 | 余额、活跃代理、即将过期、流量摘要、快捷购买 |
| `/static/buy` | 静态家宽购买 | 用途、IP 类型、地区、线路、协议、时长、数量、价格预览、余额支付 |
| `/static/proxies` | 已购 IP 列表 | 筛选、状态、复制、续费、刷新 IP、刷新凭据、导出 |
| `/static/proxies/:id` | 代理详情 | 凭据、流量、生命周期、操作记录、Webhook 投递 |
| `/account/recharge` | 账户充值 | 余额、充值金额、易支付、充值状态 |
| `/billing` | 计费流水 | 钱包流水、订单号、金额、状态、分页 |
| `/developer/apps` | 开发者 App | 多 App、AppID、Secret、IP 白名单、回调地址、启停 |
| `/tools/proxy-validator` | 代理验证工具 | HTTP/SOCKS5 验证、批量输入、结果导出 |
| `/coupons` | 优惠券 | 领取、可用、已用、过期 |
| `/referrals` | 推广账户 | 邀请码、佣金、提现入口、推广链接 |
| `/feedback` | 反馈建议 | 分类、标题、内容、联系方式、历史记录 |
| `/settings` | 账户设置 | 资料、安全设置、语言、主题 |

用户面板的第一屏必须让客户知道余额、活跃代理和下一步动作；购买页必须让客户能直接下单，不需要阅读说明。

## 4. 用户故事

### U1：新客户注册并完成首充

作为新客户，我希望注册后马上看到余额和充值入口，这样我可以先充值再购买静态家宽。

验收：

- 注册成功后进入概览页。
- 顶部和概览卡片都显示余额。
- 充值页可选择固定金额或输入自定义金额。
- 易支付回调成功后余额更新，重复回调不重复入账。
- 充值失败时展示支付状态，不影响钱包事实。

### U2：客户购买一个可用的静态家宽代理

作为客户，我希望只看到真实可售的地区、线路和数量，这样我下单后能直接拿到可用代理。

验收：

- 购买页只展示通过健康、库存、价格、能力校验的选项。
- 价格随用途、IP 类型、地区、线路、协议、时长、数量变化即时更新。
- 余额不足时引导充值，不创建无效订单。
- 提交订单后进入处理中状态，Runtime 未确认前不展示凭据。
- Runtime 确认后展示 IP、端口、协议、账号、密码，并支持复制。

### U3：客户管理已购买代理

作为客户，我希望在一个列表中管理所有静态家宽代理，这样我可以快速复制、续费、刷新和导出。

验收：

- 列表支持状态、地区、线路、协议、到期时间筛选。
- 每行展示 IP、端口、协议、地区、线路、IP 类型、到期时间、已用流量。
- 支持复制连接信息、复制账号密码、导出。
- 支持续费、IP 刷新、凭据刷新。
- 刷新类操作必须 Runtime 确认后才展示新 IP 或新凭据。

### U4：客户查看代理详情和流量

作为客户，我希望看到每个代理的使用状态和流量，这样我可以判断是否需要续费或调整使用。

验收：

- 详情页展示凭据、状态、到期时间、累计上下行、近期流量趋势。
- 停用、过期、退款状态不能继续显示为可用。
- 生命周期操作有明确结果：成功、处理中、失败退款或失败保留原状态。
- 客户看不到内部任务错误栈。

### U5：客户使用开发者 App 自动化管理

作为有自动化需求的客户，我希望创建多个 App 并分别配置密钥、白名单和回调，这样不同业务系统可以隔离使用。

验收：

- 可以创建、停用、删除多个 App。
- AppSecret 只在创建或重置时完整展示。
- 支持 IP 白名单和回调地址。
- 开发者 API 支持查询代理、查询流量、续费、刷新 IP、刷新凭据。
- 请求签名、防重放、Webhook 签名和重试状态可验证。

### U6：客户使用优惠券、推广和反馈

作为客户，我希望优惠、推广和反馈入口简单直接，不影响购买主流程。

验收：

- 优惠券可领取、查看、使用、过期。
- 推广页展示邀请码、推广链接、佣金比例、余额和提现入口。
- 反馈页支持分类、标题、内容和联系方式。
- 这些页面不阻塞购买和代理管理主流程。

## 5. 管理面板信息架构

建议路由：

| 路由 | 页面 | V1 能力 |
|---|---|---|
| `/admin` | 管理概览 | 销售、余额、活跃订单、节点健康、库存低水位 |
| `/admin/users` | 用户 | 用户列表、详情、余额、订单、开发者 App |
| `/admin/wallets` | 钱包和充值 | 钱包流水、充值订单、回调状态、人工调整审计 |
| `/admin/products` | 产品和价格 | 用途、IP 类型、地区、线路、时长、价格、速率策略 |
| `/admin/inventory` | IP 库存 | 可售、冻结、已售、禁用、质量差、退役 |
| `/admin/orders` | 订单 | 订单、代理账号、生命周期、流量、任务时间线 |
| `/admin/nodes` | 家宽节点 | 节点列表、健康、能力、可售原因、退役 |
| `/admin/nodes/realtime` | 实时节点面板 | WebSocket 实时速度、CPU、RAM、连接数、趋势 |
| `/admin/nodes/:id/terminal` | Web SSH | 权限校验、PTY、审计、超时 |
| `/admin/runtime-jobs` | 任务控制台 | 发货、续费、停用、刷新、恢复、对账、DLQ |
| `/admin/notifications` | 通知 | Bark、Webhook、通知事件、投递记录 |
| `/admin/coupons` | 优惠券 | 发放、领取、使用、过期 |
| `/admin/referrals` | 邀请规则 | 比例、结算、提现审核 |
| `/admin/feedback` | 反馈 | 客户反馈、处理状态 |
| `/admin/audit` | 审计日志 | 钱包、库存、节点、SSH、Runtime 操作 |
| `/admin/settings` | 系统设置 | 站点配置、支付配置、安全配置 |

管理面板优先展示商业运营状态，再允许下钻到 Runtime 和任务细节。

## 6. 管理员故事

### A1：管理员配置可售产品

作为管理员，我希望配置用途、地区、线路、价格、库存和速率策略，这样客户看到的购买选项是真实可售的。

验收：

- 禁用产品、线路、库存后用户购买页不展示。
- 节点不健康、Bundle 不兼容、Runtime digest 不一致时库存不可售。
- 价格变更不影响已下单订单的金额快照。
- 每个产品绑定限速和连接数策略。

### A2：管理员查看节点真实状态

作为管理员，我希望看到所有家宽节点的实时和历史状态，这样可以主动退役异常节点。

验收：

- 实时面板展示在线状态、速度、CPU、RAM、存储、连接数、流量、最后同步。
- 支持实时模式和 5 分钟、1 小时、24 小时、7 天历史模式。
- 节点不可售原因可见。
- 支持详情、Web SSH、禁止售卖、退役、重启 Runtime、触发对账。

### A3：管理员处理订单和生命周期

作为管理员，我希望从订单详情看到业务动作、钱包、库存、Runtime apply 的完整时间线，这样异常时可以安全判断。

验收：

- 订单详情展示钱包冻结/扣款、库存预约、代理账号、Runtime apply、通知投递。
- 支持安全重试、取消、对账和标记人工处理。
- 不能通过手工改状态绕过钱包、库存和 Runtime 状态机。
- 所有敏感操作写审计日志。

### A4：管理员接入和恢复节点

作为管理员，我希望能一键安装、Web SSH、自动恢复 NodeAgent，这样节点运维不需要登录多套系统。

验收：

- 创建节点时可填 SSH Key 或密码、IP、端口、root/用户。
- 可复制一键安装脚本，也可由平台 SSH 自动安装。
- NodeAgent 重装后按 Postgres 期望状态恢复已有订单。
- SSH 自动恢复受 host key、并发、冷却、审计限制。

## 7. 任务映射

| 交付任务 | 前端验收重点 |
|---|---|
| T1 薄工程基线 + 第一台节点在线 | 用户壳、管理壳、节点在线列表必须采用设计 token 和导航骨架 |

## 8. 前端完成定义

每个涉及前端的 Task 完成时，必须满足：

- 页面路由、加载、空状态、错误状态、成功状态齐全。
- 表格支持必要筛选、分页、复制和刷新。
- 金额、时间、状态、协议、流量单位展示一致。
- 用户面板不出现内部技术状态。
- 管理面板敏感操作有确认、权限和审计入口。
- 桌面端先验收，移动端至少保证登录、充值、购买、复制凭据、续费可用。
- 与 `frontend_design/` 的页面结构和信息密度对照通过。
